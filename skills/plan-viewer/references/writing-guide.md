# writing-guide: plan-viewer 文章ガイド

## 1. 理解の深さガイドライン（Why）

plan-viewer の目的は「抽象的なまとめを読む」ではなく「実装を本当に理解する」こと。
読者（非エンジニア）が以下を達成できるレベルで書く:

- **なぜこの変更が必要か**を自分の言葉で説明できる
- **何がどう変わるのか**を具体的にイメージできる（ファイル名・役割の解説付き）
- **なぜこのアプローチなのか**を理解できる（他の選択肢との比較があれば尚良い）
- **変更しなかったらどうなるか**がわかる

避けるべきパターン:
- X 「認証機能を改善した」→ 何がどう改善されたか不明
- O 「ログイン時にパスワードの正しさを検証する処理（`validateToken` 関数）にバグがあり、期限切れのトークンでもログインできてしまっていた。これを修正して、期限切れの場合は再ログインを求めるようにした。修正しないと、退職した社員のアカウントで引き続きアクセスできてしまう」

推奨パターン（各セクションの深さの目安）:
- 「一言でいうと」→ 30字で本質を掴む
- 「なぜこれが必要なのか」→ **現状の問題 + 放置した場合のリスク**まで書く
- 「何をしたのか」→ **具体的な変更内容 + 選択理由**。ファイル名が出たら役割を括弧解説
- 「ステップ」→ **各ステップの目的を添える**（手順だけ列挙しない）
- 「ポイント」→ **判断に迷ったこと・見送ったこと**を含める（読者が「全体像を掴めている」感覚を得られる）

## 2. コンテンツ構成テンプレート（What）

```
一言でいうと（30文字以内で本質）
なぜこれが必要なのか（現状の問題 + 放置した場合のリスク）
何をしたのか / 何をするのか（具体的な変更 + 選択理由）
ステップ / 進捗（各ステップの目的付き）
ポイント・補足（見送った選択肢・判断の迷い・理解の確認ポイント）
```

## 3. 文章ルール（How）

各ルールに Why を添える:

1. **専門用語は括弧で解説 + 用語集リンク** — 初見の読者がつまずかないため
   - O: `<a href="glossary.html#api">API</a>（プログラム同士が会話する窓口）`
2. **括弧解説の後に「なぜ必要か」を1文で補足** — 用語の意味がわかっても「なぜここで使うのか」がわからないと全体像が掴めないため
   - O: 「API（プログラム同士が会話する窓口）を追加する。これがないとフロントエンドからデータを取得できない」
3. **比喩→実際の概念への橋渡し** — 比喩だけだと何をしているか分からず次のアクションが取れないため
   - X: 「hookは自動ドアのセンサーみたいなもの」
   - O: 「hookは自動ドアのセンサーみたいなもの。コードの特定の処理が実行される直前・直後に、別の処理を差し込める仕組み」
4. **ファイル名・関数名には役割を括弧解説** — 名前だけでは非エンジニアにとって意味不明なため
   - O: 「`auth.ts`（ログイン認証を担当するファイル）の `validateToken`（トークンが正しいか検証する関数）を修正する」
   - X: 「`auth.ts` の `validateToken` を修正する」
5. **操作文に目的を添える** — 操作だけだと「なぜこの変更が必要か」が伝わらず優先度を判断できないため
   - O: 「dependency array（変更を監視する対象リスト）に `userId` を追加する。これにより、ユーザーが切り替わったときに画面が正しく更新される」
   - X: 「dependency arrayに `userId` を追加する」
6. **実装は概念レベルで、ただし理解できる深さで** — コードスニペット自体はノイズだが、処理の流れや判断の根拠は必要
   - O: 「新しいAPIエンドポイント（受付窓口）を追加する」
   - X: 「`app.get('/api/users', ...)` を追加する」
7. **変更を伴う plan では diff box で Before/After を示す** — 視覚的な差分が理解を促進するため
   - 赤背景 + 取り消し線 = 削除、緑背景 + 太字 = 追加。CSS クラスは `page-template.md` の diff セクション参照

## 4. 用語集ルール

- ファイル: `${PLAN_VIEWER_DIR}/glossary.html` — 新用語は `<dl>` のアルファベット順に追加、`id` 属性つき
- 構造: **ひとことで**（20字以内）→ **もう少し詳しく**（2-3文）→ **このプロジェクトでは**（該当時のみ）
- 既存の `~/plan-viewer/glossary.html` 自体がテンプレート
